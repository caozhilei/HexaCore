# 沙箱（Sandbox）层技术方案文档

## 1. 概述

### 1.1 定位与职责

沙箱层是六元组架构中的安全隔离与权限控制层，负责为多Agent环境提供企业级的安全保障。基于HexaCore安全模型和Docker沙箱实现，沙箱层的核心职责包括：

- **多层级安全隔离**：通过Sandbox Mode、Tools Policy、Elevated Mode三层防护机制，实现从容器级到工具级的全面安全控制
- **容器化执行环境**：为每个Agent或会话提供独立的Docker容器，确保故障隔离和资源隔离
- **细粒度工具权限**：实现7层工具权限过滤链，支持从全局到Agent级的精细化权限控制
- **权限提升管理**：基于发送者白名单和审批流程，实现高危操作的权限提升机制
- **安全审计与监控**：记录所有安全相关操作，提供完整的审计日志和实时监控告警

### 1.2 在六元组架构中的位置

沙箱层作为六元组架构的安全屏障，承担着"安全守门员"的角色：

```
技能执行请求 → 沙箱层 → 三层安全检查 → 沙箱内执行 → 结果返回
```

与六元组其他元组的协作关系：
- **与技能层协作**：为技能执行提供安全隔离的沙箱环境，防止技能间的相互影响
- **与频道层协作**：根据Session Key确定沙箱策略，确保不同工作区的安全隔离
- **与记忆层协作**：保护记忆数据不被未授权访问，实现安全的数据读写
- **与出入口层协作**：验证输入来源的可信度，执行渠道级安全策略
- **与路由层协作**：根据路由规则应用不同的沙箱策略，支持信任链传递

### 1.3 设计目标

基于HexaCore框架的设计目标：

1. **HexaCore安全模型对齐**：完全遵循HexaCore安全架构，实现三层防护机制的无缝集成
2. **企业级安全合规**：满足金融、医疗等行业的安全合规要求，提供完整的审计追踪能力
3. **高性能低开销**：通过容器复用和资源动态分配，确保沙箱隔离的性能开销<10%
4. **灵活的配置策略**：支持三种沙箱模式和7层权限过滤链的灵活配置，适应不同信任级别的Agent
5. **可靠的高可用性**：实现容器故障自动恢复和安全降级机制，确保系统持续可用

## 2. HexaCore参考架构

### 2.1 三层防护架构

基于HexaCore安全模型，沙箱层采用三层防护架构，逐层递进地保障系统安全：

```
第一层：Sandbox Mode（容器隔离层）
├── off模式：无沙箱，直接运行在宿主机（仅限系统级Agent）
├── non-main模式：非主会话容器化，主会话在宿主机（默认模式）
└── all模式：完全容器化，所有会话都在独立容器（高安全需求）

第二层：Tools Policy（工具权限策略层）
├── 7层过滤链：从全局到Agent级的精细化权限控制
├── 风险分级：低风险工具自动放行，高风险工具触发审批
└── 白名单机制：可信工具和来源的直接授权

第三层：Elevated Mode（权限提升层）
├── 审批流程：多级审批支持，可配置审批人和超时时间
├── 临时授权：权限提升有时效性，到期自动撤销
└── 审计追踪：完整记录权限提升的申请、审批和执行过程
```

## 3. 核心组件设计

### 3.1 容器隔离层（Sandbox Mode）

容器隔离层基于Docker实现，提供三种沙箱模式以适应不同安全需求：

**off模式**：无沙箱隔离，直接运行在宿主机环境。仅限系统级Agent和高度信任的内部服务使用。

**non-main模式**：默认沙箱模式。主会话（Main Session）运行在宿主机，非主会话（如技能执行、数据查询等）运行在独立容器中。平衡了性能和安全需求。

**all模式**：完全容器化。所有会话都在独立容器中运行，提供最高级别的安全隔离。适用于第三方技能、不受信任的Agent或处理敏感数据的场景。

核心算法实现：
```typescript
class SandboxModeManager {
  async createSandbox(
    agentId: string,
    sessionKey: string,
    mode: SandboxMode
  ): Promise<SandboxContext> {
    switch (mode) {
      case 'off':
        // 无沙箱，直接返回宿主机环境上下文
        return this.createHostContext(agentId, sessionKey);
      case 'non-main':
        // 非主会话容器化
        return this.createNonMainContainer(agentId, sessionKey);
      case 'all':
        // 完全容器化
        return this.createFullContainer(agentId, sessionKey);
      default:
        throw new Error(`Unsupported sandbox mode: ${mode}`);
    }
  }
}
```

### 3.2 工具权限策略层（Tools Policy）

工具权限策略层实现7层过滤链，确保每个工具调用都经过多层安全检查：

1. **Tool Profile过滤**：基于HexaCore工具组定义，检查工具是否在允许的工具组内
2. **Provider Tool Profile过滤**：遵循HexaCore Provider级策略，检查Provider级别的工具权限
3. **Global Tool Policy过滤**：基于HexaCore.json全局配置，应用系统级的工具策略
4. **Provider Tool Policy过滤**：与HexaCore Provider认证模型对齐，检查Provider认证状态
5. **Agent-specific Tool Policy过滤**：基于HexaCore Agent配置，应用Agent级别的个性化策略
6. **Agent Provider Policy过滤**：遵循HexaCore Agent级Provider设置，检查Agent与Provider的绑定关系
7. **Sandbox Tool Policy过滤**：与HexaCore Docker沙箱权限控制集成，检查容器级别的工具权限

核心算法实现：
```typescript
class ToolPermissionFilter {
  async filterToolAccess(
    toolName: string,
    context: SecurityContext,
    config: SecurityConfig
  ): Promise<FilterResult> {
    const filters = [
      // 1. Tool Profile - 基于HexaCore工具组定义
      this.filterByToolProfile.bind(this),
      // 2. Provider Tool Profile - HexaCore Provider级策略
      this.filterByProviderProfile.bind(this),
      // 3. Global Tool Policy - HexaCore.json全局配置
      this.filterByGlobalPolicy.bind(this),
      // 4. Provider Tool Policy - HexaCore Provider认证模型
      this.filterByProviderPolicy.bind(this),
      // 5. Agent-specific Tool Policy - HexaCore Agent配置
      this.filterByAgentPolicy.bind(this),
      // 6. Agent Provider Policy - HexaCore Agent级Provider设置
      this.filterByAgentProviderPolicy.bind(this),
      // 7. Sandbox Tool Policy - HexaCore Docker沙箱权限控制
      this.filterBySandboxPolicy.bind(this)
    ];
    
    for (const filter of filters) {
      const result = await filter(toolName, context, config);
      if (result.action !== 'continue') {
        return result;
      }
    }
    
    return { action: 'allow', reason: 'passed_all_filters' };
  }
}
```

### 3.3 权限提升层（Elevated Mode）

权限提升层实现基于发送者的权限提升机制，支持白名单自动授权和审批流程：

**白名单机制**：系统管理员、可信用户和可信工具可以加入白名单，获得特定权限的自动授权。

**审批流程**：高危操作触发多级审批流程，支持即时审批和延迟审批两种模式。

**审计追踪**：所有权限提升操作都被完整记录，包括申请人、审批人、操作内容、时间戳等信息。

核心算法实现：
```typescript
class ElevationManager {
  async requestElevation(
    request: ElevationRequest,
    config: ElevationConfig
  ): Promise<ElevationResult> {
    // 1. 检查白名单（HexaCore配对机制）
    if (await this.isWhitelisted(request.requester, request.tool)) {
      return { approved: true, reason: 'whitelisted', expiresIn: 3600 };
    }
    
    // 2. 启动审批流程
    const approvalId = await this.startApprovalFlow(request, config);
    
    // 3. 设置超时处理
    const timeoutPromise = this.waitForTimeout(config.timeout);
    const approvalPromise = this.waitForApproval(approvalId);
    
    // 4. 等待结果
    const result = await Promise.race([approvalPromise, timeoutPromise]);
    
    if (result === 'timeout') {
      // 5. 超时自动拒绝
      await this.recordDenial(request, 'timeout');
      return { approved: false, reason: 'timeout' };
    }
    
    // 6. 记录审计日志（HexaCore操作追踪）
    await this.recordApproval(request, result);
    
    return {
      approved: result.approved,
      reason: result.reason,
      expiresIn: result.expiresIn || 300,
      approver: result.approver
    };
  }
}
```

## 4. 安全策略实现

### 4.1 白名单机制

白名单机制基于HexaCore配对机制实现，支持多维度白名单配置：

**用户白名单**：可信用户ID列表，这些用户可以执行特定工具而无需审批。

**工具白名单**：低风险工具列表，这些工具可以被任何用户执行。

**来源白名单**：可信IP地址、域名或渠道列表，从这些来源的请求自动获得信任。

白名单配置示例：
```json
{
  "whitelist": {
    "users": ["admin", "system", "vip-agent-001"],
    "tools": ["read", "file_read", "weather_query"],
    "sources": ["192.168.1.0/24", "api.openai.com", "official-wechat"],
    "expiry": 86400
  }
}
```

### 4.2 审批流程设计

审批流程支持两种模式，适应不同的安全需求：

**即时审批模式**：审批人必须在5分钟内响应，适用于需要快速决策的场景。支持单个审批人或审批组。

**延迟审批模式**：审批人可以在1小时内响应，支持多人审批和法定人数要求。适用于高风险操作的审慎决策。

审批流程状态机：
```
申请提交 → 审批分配 → 审批处理 → (批准/拒绝) → 执行/终止
    ↓          ↓          ↓
  超时检测   提醒通知   审计记录
```

### 4.3 审计日志记录

审计日志系统记录所有安全相关操作，确保完整的安全追溯能力：

**审计事件类型**：
- 容器创建/销毁事件
- 工具权限检查结果
- 权限提升申请与审批
- 安全策略违反事件
- 异常行为检测事件

## 5. 性能优化策略

### 5.1 容器复用

为减少容器启动开销，实现容器复用机制：

**相同配置复用**：相同Agent配置、相同沙箱模式的请求复用现有容器。

**会话级复用**：同一会话的多次技能调用在同一个容器中执行。

**空闲容器池**：维护一个空闲容器池，预创建常用配置的容器实例。

### 5.2 资源动态分配

基于实时负载动态调整容器资源配额：

**CPU动态分配**：根据容器内任务的计算复杂度动态调整CPU配额。

**内存弹性扩展**：监控容器内存使用情况，在接近限制时自动扩展。

**网络带宽控制**：根据任务类型调整网络带宽，避免资源争抢。

### 5.3 快速恢复

确保容器故障时的快速恢复能力：

**健康检查机制**：定期检查容器健康状态，及时发现故障容器。

**自动重建策略**：故障容器自动重建，保留必要的会话状态。

**状态恢复机制**：从检查点恢复容器状态，减少中断时间。

## 6. 容错处理

### 6.1 故障隔离

确保容器故障不影响其他容器和宿主机：

**资源隔离**：每个容器有独立的cgroup命名空间，防止资源泄漏。

**网络隔离**：容器间网络隔离，防止网络攻击传播。

**文件系统隔离**：只读根文件系统，防止恶意文件写入。

### 6.2 资源限制

防止单个容器耗尽系统资源：

**内存硬限制**：每个容器有严格的内存上限，超限即终止。

**CPU配额限制**：限制容器的CPU使用比例，确保公平调度。

**磁盘IO限制**：限制容器的磁盘读写速度，防止IO风暴。

### 6.3 安全审计

确保所有操作可追溯、可审计：

**不可篡改日志**：审计日志使用数字签名，防止篡改。

**实时告警**：检测到可疑操作时实时告警通知。

**定期审计报告**：生成定期的安全审计报告，用于合规检查。
