# 基于HexaCore框架的记忆层技术方案文档

## 1. 概述

### 1.1 记忆层在六元组架构中的定位
记忆层是六元组架构中的第五个元组，承担持久化知识管理和上下文优化的核心职责。在HexaCore框架中，记忆层通过三层优化机制（Compaction、Pruning、Memory Sidecar）解决长期对话中的上下文膨胀和成本控制问题，确保智能体在保持对话连续性的同时大幅降低Token消耗。

### 1.2 三层优化机制概览
基于HexaCore记忆系统的三层优化机制包括：

1. **Compaction（压缩）** - 自动识别上下文窗口接近上限时，将旧对话摘要化，保留关键信息，基于HexaCore的T5/BART摘要算法实现。
2. **Session Pruning（修剪）** - 针对工具结果进行智能修剪，优化缓存命中率，基于HexaCore TTL机制实现，降低Anthropic缓存成本60%。
3. **Memory Sidecar（向量存储）** - 独立进程服务，持久化长期记忆，基于SQLite存储和OpenAI Embedding向量实现语义检索。

### 1.3 核心性能目标
- **Token消耗降低**：≥96%（对比无优化场景）
- **响应时间降低**：≥68%（通过缓存优化和并行处理）
- **缓存命中率提升**：≥80%（通过智能修剪策略）
- **对话连续性保持**：关键信息保留率≥90%

## 3. 核心算法实现

### 3.1 Compaction（压缩）算法

#### 3.1.1 算法流程
```typescript
class ConversationCompactor {
  async compactConversation(
    sessionKey: string,
    conversationHistory: ConversationTurn[],
    config: CompactionConfig
  ): Promise<CompactionResult> {
    // 1. 检查是否需要压缩（窗口接近上限）
    if (!this.needsCompaction(conversationHistory, config.threshold)) {
      return { action: 'skip', reason: 'below_threshold' };
    }
    
    // 2. 识别可压缩的对话区间
    const compressionWindow = this.identifyCompressionWindow(
      conversationHistory,
      config.windowSize
    );
    
    // 3. 生成摘要（基于HexaCore T5/BART摘要算法）
    const summary = await this.generateSummary(
      compressionWindow,
      config.summaryModel
    );
    
    // 4. 质量评估
    const qualityScore = await this.evaluateSummaryQuality(
      summary,
      compressionWindow
    );
    
    // 5. 应用压缩
    const compressedHistory = this.applyCompaction(
      conversationHistory,
      compressionWindow,
      summary
    );
    
    return {
      action: 'compacted',
      summary,
      qualityScore,
      tokenSavings: this.calculateTokenSavings(compressionWindow, summary),
      compressedHistory
    };
  }
}
```

#### 3.1.2 关键子算法
1. **上下文窗口使用率检测**：
   ```typescript
   needsCompaction(history: ConversationTurn[], threshold: number): boolean {
     const totalTokens = this.calculateTotalTokens(history);
     const maxTokens = config.maxContextTokens; // 如Claude Opus: 200K
     return (totalTokens / maxTokens) >= threshold; // 默认0.8
   }
   ```

2. **压缩窗口识别**：
   ```typescript
   identifyCompressionWindow(history: ConversationTurn[], windowSize: number): ConversationTurn[] {
     // 选择历史记录中最早且连续的windowSize轮对话
     const startIdx = Math.max(0, history.length - windowSize - 10); // 保留最近10轮作为活跃上下文
     return history.slice(startIdx, startIdx + windowSize);
   }
   ```

3. **摘要质量评估**：
   ```typescript
   evaluateSummaryQuality(summary: string, original: ConversationTurn[]): number {
     // 基于ROUGE/BLEU分数计算摘要与原文的关键信息重叠度
     const rougeScore = calculateRougeL(summary, original.map(t => t.content));
     const coverageScore = calculateCoverage(summary, extractKeyPoints(original));
     return (rougeScore * 0.6 + coverageScore * 0.4);
   }
   ```

### 3.2 Session Pruning（修剪）算法

#### 3.2.1 算法流程
```typescript
class SessionPruner {
  async pruneSession(
    sessionKey: string,
    sessionData: SessionData,
    config: PruningConfig
  ): Promise<PruningResult> {
    // 1. 识别低价值内容（基于频率、时间、相关性）
    const lowValueSegments = this.identifyLowValueContent(
      sessionData,
      config.valueThreshold
    );
    
    // 2. 软修剪（保留关键信息）
    if (config.mode === 'soft') {
      const retained = this.softPrune(sessionData, lowValueSegments);
      return {
        action: 'soft_pruned',
        segmentsRemoved: lowValueSegments.length,
        tokenReduction: this.calculateTokenReduction(lowValueSegments),
        retainedData: retained
      };
    }
    
    // 3. 硬清除（完全替换）
    const replaced = this.hardReplace(sessionData, lowValueSegments);
    return {
      action: 'hard_replaced',
      segmentsRemoved: lowValueSegments.length,
      tokenReduction: this.calculateTokenReduction(lowValueSegments),
      replacedData: replaced
    };
  }
}
```

#### 3.2.2 价值评估指标
1. **时间衰减因子**：`decay(t) = exp(-λ * (t_now - t_event))`，λ=0.001
2. **使用频率**：过去N次对话中引用的次数，N=100
3. **相关性得分**：基于语义相似度的向量匹配，阈值=0.7
4. **工具结果重要性**：工具类型权重（查询=0.8，写入=0.5，执行=0.3）

#### 3.2.3 修剪策略配置
```typescript
interface PruningConfig {
  mode: 'soft' | 'hard';           // 软修剪保留头尾，硬清除完全替换
  ttl: number;                     // 生存时间（秒），默认86400（24小时）
  cacheOptimization: {
    enabled: boolean;              // 缓存优化开关
    targetHitRate: number;         // 目标缓存命中率，默认0.8
    evictionPolicy: 'lru' | 'lfu'; // 淘汰策略，默认LRU
  };
  costReductionTarget: number;     // 成本降低目标，默认0.6（60%）
}
```

### 3.3 Memory Sidecar（向量存储）算法

#### 3.3.1 知识提取算法
```typescript
class KnowledgeExtractor {
  async extractKeyInformation(
    conversationHistory: ConversationTurn[]
  ): Promise<ExtractedKnowledge[]> {
    // 1. 用户偏好提取（基于频率和确认语句）
    const preferences = this.extractUserPreferences(conversationHistory);
    
    // 2. 承诺事项识别（基于承诺关键词和时间表达）
    const commitments = this.extractCommitments(conversationHistory);
    
    // 3. 重要事实抽取（基于实体识别和关系提取）
    const importantFacts = this.extractImportantFacts(conversationHistory);
    
    // 4. 向量化存储
    const embeddings = await this.generateEmbeddings([
      ...preferences.map(p => p.text),
      ...commitments.map(c => c.text),
      ...importantFacts.map(f => f.text)
    ]);
    
    return this.packageKnowledge(
      preferences,
      commitments,
      importantFacts,
      embeddings
    );
  }
}
```

#### 3.3.2 语义检索算法
```typescript
class SemanticSearchEngine {
  async searchRelevantMemory(
    query: string,
    sessionKey: string,
    config: SearchConfig
  ): Promise<SearchResult[]> {
    // 1. 查询向量化
    const queryEmbedding = await this.embedQuery(query);
    
    // 2. 向量相似度计算（余弦相似度）
    const similarities = await this.calculateSimilarities(
      queryEmbedding,
      sessionKey
    );
    
    // 3. 阈值过滤和重排序
    const filtered = similarities.filter(s => s.score >= config.similarityThreshold);
    const reranked = config.rerankEnabled ? 
      await this.rerankWithCrossEncoder(filtered, query) : filtered;
    
    // 4. 返回TopK结果
    return reranked
      .sort((a, b) => b.score - a.score)
      .slice(0, config.topK);
  }
}
```

#### 3.3.3 边车同步机制
1. **增量更新策略**：每次对话结束后，仅提取新增内容的关键信息
2. **批量处理优化**：每10轮对话进行一次批量向量化，减少API调用
3. **一致性保证**：采用WAL（Write-Ahead Logging）机制，确保故障恢复后数据一致

## 4. Token成本优化模型

### 4.1 成本优化计算模型

#### 4.1.1 无优化场景Token消耗
```
T_total = T_context + T_completion
T_context = Σ(t_i)  (i=1 to n)  // 历史对话Token累计
T_completion = f(response_length)  // 生成响应Token

示例：100轮对话，平均每轮2000 tokens
T_context = 100 × 2000 = 200,000 tokens
T_completion = 4,000 tokens (平均响应长度)
T_total = 204,000 tokens
成本 ≈ 204,000 × $0.00002 = $4.08 (GPT-4o定价)
```

#### 4.1.2 Compaction优化效果
```
压缩比例: r_c = 0.2 (保留20%关键信息)
压缩后Token: T_compacted = T_context × r_c = 200,000 × 0.2 = 40,000 tokens
Token节约: ΔT_c = T_context - T_compacted = 160,000 tokens
节约比例: 160,000 / 200,000 = 80%
```

#### 4.1.3 Pruning优化效果
```
缓存命中率提升: Δh = 0.6 (60%)
工具结果平均长度: L_tool = 800 tokens
每轮对话平均工具调用: C_tool = 2
总工具结果Token: T_tools = n × C_tool × L_tool = 100 × 2 × 800 = 160,000 tokens
修剪后保留: T_pruned = T_tools × (1 - Δh) = 160,000 × 0.4 = 64,000 tokens
Token节约: ΔT_p = 96,000 tokens
```

#### 4.1.4 综合优化效果
```
优化前总Token: T_before = 204,000 + 160,000 = 364,000 tokens
优化后总Token: T_after = 40,000 + 64,000 + 4,000 = 108,000 tokens
总Token节约: ΔT_total = 364,000 - 108,000 = 256,000 tokens
节约比例: 256,000 / 364,000 ≈ 70.3%
实际成本降低: (364,000 - 108,000) / 364,000 × 100% = 70.3%
```

## 5. 性能优化策略

### 5.1 增量压缩策略
- **触发条件**：上下文窗口使用率≥80%且新增对话≥5轮
- **压缩粒度**：每次压缩固定窗口大小（默认20轮）
- **质量保证**：摘要质量评分≥0.7才应用压缩

### 5.2 智能缓存优化
1. **分级缓存策略**：
   - L1缓存：内存缓存，存储高频访问记忆（TTL=300秒）
   - L2缓存：Redis缓存，存储中频记忆（TTL=3600秒）
   - L3存储：SQLite向量存储，持久化长期记忆

2. **预取机制**：
   - 基于会话模式预测可能需要的记忆片段
   - 异步加载相关向量，减少检索延迟

### 5.3 批量处理与并行化
1. **向量化批量处理**：每10轮对话批量生成embedding，减少API调用
2. **并行压缩**：多个会话的压缩操作并行执行，利用多核CPU
3. **异步索引**：记忆写入后异步构建索引，不影响主流程响应

### 5.4 资源动态分配
- **内存配额**：根据会话活跃度动态分配缓存空间
- **CPU优先级**：高优先级会话的压缩操作优先调度
- **IO优化**：采用WAL机制减少磁盘写入次数

## 6. 容错处理机制

### 6.1 数据一致性保证
1. **事务支持**：所有记忆操作支持ACID事务
2. **幂等设计**：重复操作不会导致数据不一致
3. **版本控制**：每次修改记录版本号，支持回滚

### 6.2 故障恢复机制
1. **压缩备份**：压缩前自动备份原始对话，支持手动恢复
2. **健康检查**：定期检查记忆服务状态，异常时自动重启
3. **灾难恢复**：支持从备份恢复数据，RTO≤15分钟，RPO≤5分钟

### 6.3 降级策略
1. **压缩降级**：压缩服务故障时，使用原始对话（无压缩）
2. **边车降级**：向量检索失败时，退回关键词匹配
3. **缓存降级**：Redis不可用时，使用内存缓存+数据库直读

### 6.4 监控与告警
1. **关键指标**：
   - 压缩成功率（目标≥95%）
   - 检索延迟P99（目标<100ms）
   - Token消耗率（目标降低≥96%）

2. **告警规则**：
   - 连续3次压缩失败
   - Token消耗突增>100,000/小时
   - 边车服务不可用>5分钟

```
2. 使用率≥80% → 触发压缩模块
3. 压缩模块生成摘要 → 质量检查 → 应用压缩
4. 同时，修剪模块分析工具结果价值 → 实施修剪
5. 边车模块提取关键知识 → 向量化存储
6. 构建优化后上下文 → 传递给Agent
7. Agent处理 → 返回响应
```

### 8.3 关键组件交互协议
1. **压缩模块 ↔ 记忆管理器**：
   - 事件：`compaction_triggered`
   - 数据：`{sessionKey, history, config}`
   - 响应：`{action, summary, qualityScore, tokenSavings}`

2. **修剪模块 ↔ 缓存系统**：
   - 事件：`prune_analysis`
   - 数据：`{sessionKey, toolResults, ttl}`
   - 响应：`{retainedData, segmentsRemoved, costReduction}`

3. **边车模块 ↔ 向量存储**：
   - 查询：`semantic_search`
   - 参数：`{query, topK, threshold}`
   - 返回：`{results: [{id, text, score, metadata}]}`
