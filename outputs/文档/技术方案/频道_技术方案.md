# 频道（Channels）层技术方案文档

## 1. 概述

### 1.1 定位与职责

频道层是六元组架构中的**上下文隔离的逻辑容器**，为每个Agent提供独立的“办公室”环境，确保不同Agent的会话历史、工作区文件、工具权限完全隔离。基于HexaCore框架的Session Key机制和Agent隔离模型实现，频道层的核心职责包括：

- **会话隔离**：为每个Agent创建独立的会话容器，基于HexaCore Session Key机制实现上下文隔离
- **工作区管理**：为每个Agent分配独立的工作区目录，支持文件系统隔离和资源配额管理
- **Agent生命周期管理**：负责Agent的创建、启动、监控、健康检查和优雅终止
- **状态同步与恢复**：确保Agent故障时能自动恢复会话状态，支持故障转移机制
- **性能优化**：通过会话池、延迟加载、压缩存储等机制降低内存和磁盘开销

### 1.2 在六元组架构中的位置

频道层位于路由层之后、技能/记忆/沙箱层之前，承担着“隔离容器”的核心角色：

```
用户请求 → 出入口层 → 路由层 → 频道层 → 技能层/记忆层/沙箱层
```

与六元组其他元组的协作关系：

- **与出入口层协作**：接收InboundMessage中的渠道和账户信息，用于Session Key生成
- **与路由层协作**：接收路由结果（目标Agent ID），建立对应Agent的会话容器
- **与技能层协作**：为Agent提供技能执行环境，确保技能在隔离的上下文中运行
- **与记忆层协作**：基于Session Key关联Agent的会话历史，支持上下文连续性
- **与沙箱层协作**：根据Agent安全等级应用相应的容器隔离策略（off、non-main、all）

### 1.3 设计目标

基于HexaCore框架的设计目标：

1. **HexaCore兼容性**：完全遵循HexaCore Session Key格式和Agent隔离模型，确保与HexaCore Gateway事件系统无缝集成
2. **强隔离性**：实现内存、文件系统、网络、权限等多维度隔离，确保Agent间互不干扰
3. **高性能与低延迟**：通过会话池和连接复用机制，确保平均会话建立时间<200ms
4. **高可用性与容错**：支持会话状态检查点、故障转移和自动恢复，确保单点故障不影响整体服务
5. **企业级可配置**：支持多种Agent类型配置（VIP、通用、技术支持等），提供灵活的隔离策略配置

### 2.2 核心组件交互流程

频道层的核心组件交互遵循以下事件驱动流程：

```
1. 路由层确定目标Agent → 触发Agent创建事件
2. 频道管理器接收事件 → 调用Agent管理器创建Agent实例
3. Agent管理器创建会话 → 调用Session Key生成器生成唯一标识
4. 工作区管理器分配工作区 → 设置文件系统隔离和资源配额
5. 沙箱层应用隔离策略 → 根据Agent安全等级配置容器
6. Agent启动成功 → 返回Session Key给路由层
7. 后续消息基于Session Key路由 → 直接定位到对应Agent会话
```

## 3. Session Key机制详细设计

### 3.1 Session Key生成规则

Session Key是频道层的核心标识符，采用确定性算法生成，确保同一会话在不同时间、不同实例中生成相同的Key：

```typescript
class SessionKeyGenerator {
  /**
   * 基于HexaCore标准生成Session Key
   * 格式：agent:{agentId}:{channel}:{accountId}:{peerType}:{peerId}
   */
  generateKey(message: InboundMessage, agentId: string): string {
    const peerType = this.getPeerType(message.peer);
    
    return `agent:${agentId}:${message.channel}:${message.accountId}:${peerType}:${message.peer.id}`;
  }
  
  /**
   * 确定Peer类型（直接消息、群组、频道等）
   */
  private getPeerType(peer: PeerInfo): string {
    if (peer.kind === 'dm') return 'direct';
    if (peer.kind === 'group') return 'group';
    if (peer.kind === 'channel') return 'channel';
    return 'unknown';
  }
  
  /**
   * 从Session Key解析各组成部分
   */
  parseKey(sessionKey: string): SessionKeyParts {
    const parts = sessionKey.split(':');
    
    return {
      agentId: parts[1],
      channel: parts[2],
      accountId: parts[3],
      peerType: parts[4],
      peerId: parts[5]
    };
  }
}
```

### 3.2 Session Key格式说明

| 组成部分 | 示例值 | 说明 |
|----------|--------|------|
| **前缀** | `agent:` | 固定前缀，标识为Agent会话 |
| **Agent ID** | `vip` | 目标Agent的唯一标识符，由路由层确定 |
| **渠道** | `whatsapp` | 消息来源渠道（WhatsApp、企业微信、Web等） |
| **账户ID** | `business` | 渠道账户标识符（企业ID、Bot ID等） |
| **Peer类型** | `direct` | 会话类型：direct（直接消息）、group（群组）、channel（频道） |
| **Peer ID** | `+8613812345678` | 对端标识符（用户ID、群组ID、频道ID等） |

### 3.3 并发会话处理

同一用户可与多个Agent同时对话，频道层通过以下机制支持并发会话：

1. **会话映射表**：维护`userId → agentId → sessionKey`的映射关系
2. **会话状态机**：每个会话独立的状态机，管理创建、活跃、挂起、销毁等状态
3. **资源隔离**：并发会话间共享系统资源，但工作区、内存上下文完全隔离
4. **负载均衡**：支持将会话分配到不同物理节点，实现水平扩展

## 4. 核心组件设计

### 4.1 频道容器（Channel Container）

频道容器是Agent运行的基础环境，提供完整的隔离保障：

```typescript
interface ChannelContainer {
  // 容器标识
  sessionKey: string;
  agentId: string;
  
  // 隔离配置
  isolationLevel: 'off' | 'non-main' | 'all';
  resourceLimits: ResourceQuota;
  networkPolicy: NetworkPolicy;
  
  // 运行时状态
  status: 'creating' | 'running' | 'paused' | 'stopped' | 'error';
  pid?: number;
  startTime?: Date;
  
  // 工作区管理
  workspace: WorkspaceManager;
  
  // 生命周期方法
  start(): Promise<void>;
  pause(): Promise<void>;
  resume(): Promise<void>;
  stop(): Promise<void>;
  destroy(): Promise<void>;
}
```

### 4.2 会话状态机（Session State Machine）

会话状态机确保Agent会话的确定性状态转换：

```
创建中 (creating)
    ↓
运行中 (running) ←→ 挂起中 (paused)
    ↓
停止中 (stopping)
    ↓
已停止 (stopped)
    ↑
错误状态 (error) → 恢复中 (recovering)
```

关键状态转换规则：

- **创建 → 运行**：容器启动成功，Agent进程就绪
- **运行 → 挂起**：手动暂停或资源限制触发
- **挂起 → 运行**：手动恢复或条件满足
- **运行 → 停止**：正常终止或超时触发
- **任何状态 → 错误**：容器故障、进程崩溃、资源超限
- **错误 → 恢复**：自动恢复机制触发，尝试重建会话

### 4.3 工作区管理（Workspace Management）

工作区为Agent提供独立的文件系统环境：

```
~/.HexaCore/workspace-{agentType}-{agentId}/
├── sessions/                    # 会话历史存储
│   └── {sessionKey}.jsonl      # 按Session Key存储的会话日志
├── skills/                     # Agent专用技能目录
│   ├── price-calculator/       # 示例技能：报价计算器
│   └── ticket-create/          # 示例技能：工单创建
├── data/                       # Agent数据存储
│   ├── cache/                  # 缓存数据
│   └── temp/                   # 临时文件
└── config/                     # Agent配置文件
    └── agent-local.json        # 本地覆盖配置
```

工作区管理器核心功能：

- **路径隔离**：每个Agent拥有独立的工作区根目录
- **配额管理**：磁盘空间、文件数、内存使用量限制
- **权限控制**：文件读写权限、执行权限、网络访问权限
- **快照支持**：支持工作区快照，用于备份和迁移

## 5. 上下文隔离策略

### 5.1 内存隔离

确保不同Agent的内存上下文完全隔离：

1. **进程级隔离**：每个Agent运行在独立的Node.js进程中
2. **内存空间分离**：进程间内存不共享，通过IPC通信
3. **垃圾回收独立**：每个进程拥有独立的V8堆内存和GC策略
4. **内存限制**：通过`--max-old-space-size`限制每个Agent进程的最大内存

### 5.2 数据分区

基于Session Key的数据存储隔离策略：

| 数据类型 | 存储策略 | 隔离粒度 |
|----------|----------|----------|
| **会话历史** | 按Session Key分文件存储 | 会话级 |
| **工作区文件** | 按Agent ID分配独立目录 | Agent级 |
| **缓存数据** | Redis按Session Key前缀分区 | 会话级 |
| **配置数据** | PostgreSQL按Agent ID分区 | Agent级 |

## 6. 性能优化策略

### 6.1 会话池（Session Pool）

常用会话保持在内存中，减少创建开销：

```typescript
class SessionPool {
  private pool: Map<string, ChannelContainer> = new Map();
  private lru: LRUCache<string> = new LRUCache(100); // 最大100个活跃会话
  
  // 获取会话（命中池则直接返回，否则创建）
  async get(sessionKey: string): Promise<ChannelContainer> {
    if (this.pool.has(sessionKey)) {
      this.lru.touch(sessionKey);
      return this.pool.get(sessionKey)!;
    }
    
    // 创建新会话
    const container = await this.createContainer(sessionKey);
    this.pool.set(sessionKey, container);
    this.lru.set(sessionKey);
    
    // 池满时清理最久未使用会话
    if (this.pool.size > this.maxSize) {
      const evictedKey = this.lru.evict();
      if (evictedKey) {
        await this.destroyContainer(evictedKey);
        this.pool.delete(evictedKey);
      }
    }
    
    return container;
  }
}
```

### 6.2 延迟加载（Lazy Loading）

历史会话按需加载，减少内存占用：

1. **会话状态分级**：
   - **热会话**：最近活跃，完全加载到内存
   - **温会话**：历史活跃，仅加载元数据
   - **冷会话**：长期未用，存储到磁盘，使用时重新加载

2. **加载策略**：
   - **按访问频率**：高频访问会话保持热状态
   - **按数据大小**：大数据量会话采用延迟加载
   - **按业务重要性**：VIP客户会话优先保持热状态

### 6.3 连接复用（Connection Reuse）

复用底层连接，降低网络开销：

- **数据库连接池**：共享PostgreSQL、Redis连接
- **HTTP连接池**：复用API服务连接
- **WebSocket保持**：长连接复用，减少握手开销

### 6.4 资源配额管理（Resource Quota Management）

防止单个Agent占用过多资源：

```typescript
interface ResourceQuota {
  // 内存限制
  memory: {
    maxRSS: string;      // 最大常驻内存（如"1g"）
    maxHeap: string;     // 最大堆内存
  };
  
  // CPU限制
  cpu: {
    shares: number;      // CPU份额（相对权重）
    quota: string;       // CPU时间配额（如"2.0"表示2个CPU核心）
  };
  
  // 存储限制
  storage: {
    diskQuota: string;   // 磁盘配额（如"10g"）
    inodeLimit: number;  // 文件数限制
  };
  
  // 网络限制
  network: {
    bandwidth: string;   // 带宽限制（如"100m"）
    connectionLimit: number; // 并发连接数限制
  };
}
```

## 7. 容错处理

### 7.1 会话恢复（Session Recovery）

Agent崩溃时自动恢复会话状态：

1. **状态检查点**：定期保存会话状态到持久存储
2. **恢复流程**：
   ```
   检测Agent崩溃 → 加载最近检查点 → 重建容器环境 → 
   恢复会话历史 → 通知上游组件 → 继续处理消息
   ```
3. **数据一致性保证**：通过WAL（Write-Ahead Logging）确保状态恢复的原子性

### 7.2 状态同步（State Synchronization）

多实例部署下的状态同步机制：

- **主从复制**：主节点负责状态更新，从节点异步复制
- **分布式共识**：关键状态通过Raft协议保证一致性
- **最终一致性**：非关键状态采用最终一致性模型

### 7.3 故障转移（Failover）

主Agent故障时自动迁移会话到备用Agent：

```typescript
class FailoverManager {
  async handleAgentFailure(failedAgentId: string): Promise<void> {
    // 1. 识别受影响会话
    const affectedSessions = await this.findSessionsByAgentId(failedAgentId);
    
    // 2. 选择备用Agent（基于负载、能力匹配等）
    const backupAgentId = await this.selectBackupAgent(failedAgentId);
    
    // 3. 迁移会话状态
    for (const session of affectedSessions) {
      await this.migrateSession(session.sessionKey, backupAgentId);
    }
    
    // 4. 更新路由表
    await this.updateRoutingTable(failedAgentId, backupAgentId);
    
    // 5. 通知监控系统
    await this.notifyMonitoring(failedAgentId, backupAgentId, affectedSessions.length);
  }
}
```

### 7.4 健康检查（Health Checking）

实时监控Agent健康状态：

- **心跳检测**：定期发送心跳包，检测Agent存活状态
- **资源监控**：监控CPU、内存、磁盘、网络使用率
- **响应时间监控**：跟踪消息处理延迟，检测性能降级
- **自动修复**：检测到异常时自动重启或迁移Agent
