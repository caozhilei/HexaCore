## 前言：2026年智能体开发的新范式

在2026年的技术版图中，智能体开发已经从"单一大模型 + 一堆工具"的玩具时代，进入了"分布式、多协作、高可靠"的生产时代。企业级智能体系统需要解决的核心问题不再是"如何让模型生成答案"，而是：

- 如何接入多渠道异构输入？
- 如何智能调度多个Agent协同？
- 如何保证长期运行的上下文连续性？
- 如何在多Agent环境下实现安全隔离？
- 如何控制长期对话的成本？
- 如何让能力模块可复用、可扩展？

**六元组架构**应运而生。它不是技术的简单堆砌，而是对生产级智能体系统本质需求的系统性抽象。


**最新演进：可视化管理与全双工驱动**
随着 `apps/web` 的全面推进，HexaCore 的六元组架构现已支持 **CLI + Web Dashboard 双引擎驱动**。基于 Next.js App Router、Supabase 和 shadcn/ui 构建的现代化多语言（i18n）管理后台，使得企业不仅能通过 JSON 配置即代码（Configuration-as-Code），更能通过可视化界面直观地管理 Agent、路由、技能与沙箱策略，极大降低了运维和运营门槛。

## 一、六元组架构：核心概念全景图

### 1.1 六元组的本质定义

| 核心原语 | 本质 | 核心问题 | 解决方式 |
|---------|------|---------|---------|
| **出入口（Entry Points）** | 统一协议转换层 | 如何消除平台差异？ | 异构输入标准化 |
| **路由（Routing）** | 多维度智能分发引擎 | 如何精准调度？ | 多级匹配规则 |
| **频道（Channels）** | 上下文隔离的逻辑容器 | 如何实现并发隔离？ | 独立工作区 + 会话键 |
| **技能（Skills）** | 声明式的能力单元 | 如何封装能力？ | 工具声明 + 依赖管理 |
| **记忆（Memory）** | 持久化知识管理系统 | 如何保持连续性？ | 压缩 + 向量存储 |
| **沙箱（Sandbox）** | 多层级安全隔离机制 | 如何保证安全？ | 容器化 + 权限矩阵 |

### 1.2 六元组架构的设计哲学

**原则一：单一职责**
- 每个元组解决一个独立维度的问题
- 出入口不关心路由逻辑，路由不关心记忆实现
- 模块间通过标准接口通信，解耦合

**原则二：渐进式增强**
- 基础功能由四个元组支撑：出入口、路由、频道、技能
- 进阶需求由两个元组增强：记忆、沙箱
- 可根据实际需求选择性部署

**原则三：配置即代码**
- 所有元组通过JSON配置驱动
- 支持热重载，无需重启服务
- 版本可控，可回滚

**原则四：可独立演进**
- 每个元组可以独立升级、替换
- 记忆层可从Compaction升级为RAG向量检索
- 沙箱层可从Docker升级为Kata Containers

## 二、元组一：出入口（Entry Points）

### 2.1 核心定位

**本质**：统一协议转换层，消除平台差异。

**解决的核心问题**：
- WhatsApp、Telegram、企业微信、Slack等平台消息格式各异
- 如何用一套逻辑处理所有渠道的输入？
- 如何新增渠道时不修改核心代码？

**设计目标**：
- 将所有异构输入标准化为统一的内部事件对象
- 提供统一的输出接口，让Agent回复能通过任意渠道返回
- 支持热插拔，新增渠道只需添加适配器

### 2.2 技术实现

**标准化流程**：
```
平台原始消息
    ↓
Channel适配器（channels/{platform}/）
    ↓
InboundMessage对象（标准化格式）
    ↓
路由层处理
```

**InboundMessage对象结构**：
```json
{
  "channel": "whatsapp",
  "accountId": "business",
  "peer": {
    "kind": "dm",
    "id": "+86138xxxxxxx",
    "metadata": {
      "tier": "premium",
      "name": "张三"
    }
  },
  "content": {
    "text": "查询订单#12345",
    "type": "text"
  },
  "timestamp": "2026-02-27T22:00:00Z"
}
```

### 2.4 安全策略

**三种DM策略**：

| 策略 | 含义 | 适用场景 | 安全级别 |
|-----|------|---------|---------|
| `allowlist` | 白名单模式 | 限制少数可信用户 | 高 |
| `pairing` | 配对模式 | 用户主动配对后才能使用 | 中 |
| `open` | 开放模式 | 公开服务，所有人可用 | 低 |

**推荐配置**：
```json
{
  "channels": {
    "whatsapp": {
      "accounts": {
        "vip": {
          "dmPolicy": "allowlist",
          "allowFrom": ["+86138xxxxxxx"]  // VIP专用，高安全
        }
      }
    },
    "telegram": {
      "accounts": {
        "support": {
          "dmPolicy": "pairing"  // 通用客服，中等安全
        }
      }
    },
    "web": {
      "accounts": {
        "public": {
          "dmPolicy": "open"  // 公开服务，低安全
        }
      }
    }
  }
}
```

## 三、元组二：路由（Routing）

### 3.1 核心定位

**本质**：多维度智能分发引擎，决定每条消息由哪个Agent处理。

**解决的核心问题**：
- VIP客户和普通客户需要不同的Agent服务
- "报价"、"技术支持"、"投诉"等不同需求需要分发给专门Agent
- 如何确保同一条消息永远路由到同一个Agent？
- 如何新增规则时不修改代码？

**设计目标**：
- 支持多维度匹配：对象、内容、来源、时间等
- 确定性路由：同一条消息永远路由到同一个目的地
- 高性能：支持热重载，无需重启服务
- 可扩展：新增规则只需添加配置

### 3.2 路由匹配优先级

**7级匹配规则**（从高到低）：
```
1. peer 精确匹配（私聊对象ID）
   └─ 示例：peer.id = "+86138xxxxxxx"

2. peer 元数据匹配（用户属性）
   └─ 示例：peer.metadata.tier = "premium"

3. content 关键词匹配（意图识别）
   └─ 示例：content.text 包含 "报价"、"价格"

4. content 语义匹配（Embedding相似度）
   └─ 示例：content.text 与"询价"向量的相似度 > 0.85

5. guild/team 匹配（组织单位）
   └─ 示例：guildId = "discord-guild-123"

6. account/channel 匹配（账户/渠道级）
   └─ 示例：channel = "whatsapp" AND accountId = "business"

7. default（兜底Agent）
   └─ 默认匹配到general Agent
```

## 四、元组三：频道（Channels）

### 4.1 核心定位

**本质**：上下文隔离的逻辑容器，每个Agent有独立的"办公室"。

**解决的核心问题**：
- 如何保证不同Agent的对话历史完全独立？
- 如何实现Agent级的权限隔离？
- 如何防止一个Agent的故障影响其他Agent？

**设计目标**：
- 会话隔离：不同Agent的对话历史完全独立
- 权限隔离：不同Agent的工具权限独立配置
- 数据隔离：不同Agent的工作区文件独立
- 故障隔离：一个Agent崩溃不影响其他Agent

### 4.2 Session Key设计

**Session Key是HexaCore的核心机制，确保会话隔离**：

**私聊格式**：
```
agent:{agentId}:{channel}:{accountId}:direct:{peerId}

示例：
agent:vip:whatsapp:business:direct:+86138xxxxxxx
agent:general:wecom:corp:direct:user123
```

**群聊格式**：
```
agent:{agentId}:{channel}:{accountId}:{peerKind}:{peerId}

示例：
agent:tech_team:discord:main:guild:tech-support
agent:marketing:telegram:support:group:product-feedback
```

**Session Key的作用**：
- 唯一标识一个会话
- 确保不同会话的上下文独立
- 支持并发对话：一个用户可以同时在多个Agent中对话

### 4.4 频道隔离的实际效果

**会话历史隔离**：
- `general` Agent只能看到通过`general`路由的对话
- `vip` Agent只能看到通过`vip`路由的对话
- 不同Agent的对话历史完全独立

### 5.4 Skill实现示例

**index.ts代码**：
```typescript
interface PriceParams {
  sku: string;
  quantity: number;
  customerTier?: 'standard' | 'premium' | 'vip';
  region?: 'CN' | 'US' | 'EU' | 'JP';
  currency?: 'CNY' | 'USD' | 'EUR' | 'JPY';
}

interface PriceResult {
  sku: string;
  quantity: number;
  unitPrice: number;
  discount: number;
  finalPrice: number;
  currency: string;
  validUntil: string;
  message?: string;
}

interface PricingConfig {
  products: {
    [sku: string]: {
      basePrice: {
        [currency: string]: number;
      };
      discounts: {
        standard: number;
        premium: number;
        vip: number;
      };
    };
  };
}

export async function calculatePrice(params: PriceParams): Promise<PriceResult> {
  // 1. 读取定价配置
  const pricingConfig = await readPricingConfig();

  // 2. 查找产品价格
  const product = pricingConfig.products[params.sku];
  if (!product) {
    throw new Error(`Product not found: ${params.sku}`);
  }

  // 3. 计算折扣
  let discount = 1.0;
  if (params.customerTier === 'vip') {
    discount = product.discounts.vip;
  } else if (params.customerTier === 'premium') {
    discount = product.discounts.premium;
  }

  // 4. 计算总价
  const currency = params.currency || 'CNY';
  const unitPrice = product.basePrice[currency];
  if (!unitPrice) {
    throw new Error(`Currency not supported: ${currency}`);
  }

  const subtotal = unitPrice * params.quantity;
  const finalPrice = subtotal * discount;

  // 5. 返回结果
  const result: PriceResult = {
    sku: params.sku,
    quantity: params.quantity,
    unitPrice,
    discount,
    finalPrice,
    currency,
    validUntil: getValidUntil(7) // 7天有效期
  };

  // 6. 添加折扣说明
  if (params.customerTier && params.customerTier !== 'standard') {
    result.message = `作为${params.customerTier.charAt(0).toUpperCase() + params.customerTier.slice(1)}客户，您享受${Math.round((1 - discount) * 100)}%的折扣优惠`;
  }

  return result;
}

// 读取定价配置
async function readPricingConfig(): Promise<PricingConfig> {
  const fs = require('fs').promises;
  const path = require('path');

  const configPath = path.join(__dirname, 'data', 'pricing.json');
  const content = await fs.readFile(configPath, 'utf-8');
  return JSON.parse(content);
}

// 计算有效期
function getValidUntil(days: number): string {
  const date = new Date();
  date.setDate(date.getDate() + days);
  return date.toISOString().split('T')[0];
}
```

## 六、元组五：记忆（Memory）

### 6.1 核心定位

**本质**：持久化知识管理系统，解决长期对话的上下文膨胀与成本控制问题。

**解决的核心问题**：
- 上下文窗口有限（Claude Opus: 200K tokens，GPT-4o: 128K tokens）
- 长时间运行的Agent会累积大量历史消息
- 直接截断会导致关键信息丢失
- 成本随历史长度线性增长

**设计目标**：
- 保持对话连续性：关键信息不丢失
- 控制成本：减少Token消耗
- 自动化管理：智能压缩，无需人工干预
- 可检索：支持长期记忆的向量检索

### 6.2 三层记忆机制

| 层级 | 机制 | 作用 | 持久化 | 延迟 |
|-----|------|------|-------|-----|
| **Compaction** | 对话摘要化 | 将旧对话压缩为摘要，减少Token消耗 | ✅ JSONL | 中等 |
| **Session Pruning** | 工具结果修剪 | 临时修剪旧的工具结果，优化缓存 | ❌ 内存 | 低 |
| **Memory Sidecar** | 向量存储 | 长期记忆的持久化与检索 | ✅ SQLite | 高 |

### 6.3 Compaction（压缩）

**工作原理**：
```
原始对话历史（10000条）：
├─ 用户: "你好"
├─ Agent: "您好！有什么可以帮您？"
├─ 用户: "查询订单12345"
├─ Agent: "订单状态：已发货"
├─ ... (9996条)
└─ 用户: "我的订单何时到达？"

触发条件：上下文窗口接近上限（如80%）
    ↓
压缩操作：
├─ 分析对话结构，识别关键转折点
├─ 前9000条压缩为摘要
└─ 最后1000条保持完整

压缩后历史（1001条）：
├─ [COMPACTED] 用户与客服进行了多轮对话，主要涉及订单查询、价格咨询、售后问题...
├─ Agent: "好的，我记录下来了。"
├─ 用户: "我的订单何时到达？"
└─ ... (最近999条)
```

**配置示例**：
```json
{
  "agents": {
    "defaults": {
      "compaction": {
        "mode": "auto",
        "targetTokens": 150000,  // 目标Token数（200K窗口的75%）
        "minMessages": 50,      // 最少消息数才触发压缩
        "keepLastMessages": 20, // 保留最近20条完整消息
        "summaryPrompt": "请用200字以内总结以下对话的核心内容，包括：1. 用户需求 2. 已解决问题 3. 待办事项"
      }
    }
  }
}
```

**Compaction的三个触发模式**：

| 模式 | 触发时机 | 适用场景 |
|-----|---------|---------|
| `auto` | 上下文窗口接近上限（默认80%） | 通用场景 |
| `manual` | 用户发送`/compact`命令 | 主动压缩 |
| `disabled` | 禁用 | 短对话场景 |

**手动压缩示例**：
```
用户指令：/compact 关注定价决策和待确认事项

Agent执行：
- 按照用户指定的压缩方向
- 将旧对话摘要化
- 在JSONL中插入COMPACTED标记

用户反馈：已压缩，历史记录从10000条减少到1200条
```

### 6.4 Session Pruning（修剪）

**工作原理**：
```
场景：TTL过期后的缓存优化

问题：
- Anthropic缓存只在TTL内有效（如1小时）
- 超过TTL后，下次请求需要重新缓存全部历史
- 缓存Write成本高昂

解决：
修剪（Session Pruning）→ 临时修剪旧的工具结果
    ↓
- 只修剪`toolResult`消息
- 用户+助手消息不变
- 缓存窗口重置
    ↓
下次请求：
- 缓存命中（因为窗口重置）
- 成本降低60%
```

**配置示例**：
```json
{
  "agents": {
    "defaults": {
      "contextPruning": {
        "mode": "cache-ttl",
        "ttl": "5m",                  // TTL过期5分钟后触发修剪
        "keepLastAssistants": 3,      // 保留最后3个助手消息
        "softTrimRatio": 0.3,        // 软修剪30%的工具结果
        "hardClearRatio": 0.5,       // 硬清除50%的工具结果
        "minPrunableToolChars": 50000,// 超过5万字符的工具结果才修剪
        "softTrim": {
          "maxChars": 4000,
          "headChars": 1500,
          "tailChars": 1500
        },
        "hardClear": {
          "enabled": true,
          "placeholder": "[Old tool result content cleared]"
        },
        "tools": {
          "allow": ["exec", "read"],  // 只修剪特定工具的结果
          "deny": ["*image*"]        // 跳过包含图像块的结果
        }
      }
    }
  }
}
```

**软修剪 vs 硬清除**：
```
工具结果（原始）：100000字符

软修剪（保留头尾）：
├─ "前1500字符..."
├─ ... [TRIMMED] (原始大小: 100000字符)
└─ "...后1500字符"
总计：约3000字符

硬清除（完全替换）：
[Old tool result content cleared]
总计：约50字符
```

## 七、元组六：沙箱（Sandbox）

### 7.1 核心定位

**本质**：多层级安全隔离机制，解决多Agent环境下的权限控制与故障隔离问题。

**解决的核心问题**：
- 不同Agent的信任级别不同（个人助手 vs 公共服务）
- Agent间需要隔离（VIP数据不应被通用Agent访问）
- 工具权限需要细粒度控制（禁止`exec`但允许`read`）
- 容器化部署增加复杂度

**设计目标**：
- 容器化隔离：每个Agent在独立的容器中运行
- 细粒度权限控制：工具级别权限管理
- 故障隔离：一个Agent崩溃不影响其他Agent
- 可审计：所有敏感操作记录日志

### 7.2 三层防护体系

| 层级 | 机制 | 作用 |
|-----|------|------|
| **Sandbox Mode** | 容器化隔离 | 控制Agent的运行环境 |
| **Tools Policy** | 7层过滤链 | 控制Agent可用的工具 |
| **Elevated Mode** | 发送者权限 | 控制敏感操作的授权 |

### 7.3 Sandbox Mode（容器化模式）

> **💡 Dashboard 演进**：通过后台的 `/sandbox` 模块，安全管理员能以图形化形式为指定 Agent 分配隔离级别（off / non-main / all）及资源上限（CPU/内存限制），并对 `Tools Policy`（工具过滤链）进行打钩式配置。

**三种模式对比**：

| 模式 | 运行位置 | 隔离级别 | 性能 | 适用场景 |
|-----|---------|---------|-----|---------|
| **off** | 主机 | 无隔离 | 最高 | 个人助手，完全信任 |
| **non-main** | 容器 | 会话级 | 中等 | 默认模式，平衡性能与安全 |
| **all** | 容器 | Agent级 | 较低 | 多Agent环境，严格隔离 |

**Scope模式对比**：

| Scope | 容器数量 | 隔离级别 | 适用场景 |
|-------|---------|---------|---------|
| **session** | 每会话一个 | 最高 | 严格隔离，多租户 |
| **agent** | 每Agent一个 | 中等 | Agent间隔离，会话共享 |
| **shared** | 全局共享 | 最低 | 性能优先，单Agent |

**配置示例**：
```json
{
  "agents": {
    "defaults": {
      "sandbox": {
        "mode": "non-main",       // 默认：非主会话容器化
        "scope": "session"         // 每个会话独立容器
```

### 7.4 Tools Policy（工具权限控制）

**7层过滤链**（从外到内，逐步收缩）：

```
1. Tool Profile（工具组）
   └─ tools.profile: "coding" → 扩展为 [exec, read, write, edit]

2. Provider Tool Profile（Provider级工具组）
   └─ tools.byProvider[openai].profile: "messaging" → 扩展为 [message]

3. Global Tool Policy（全局工具策略）
   └─ tools.allow: ["read", "exec"]
   └─ tools.deny: ["write", "browser"]

4. Provider Tool Policy（Provider级策略）
   └─ tools.byProvider[openai].allow: ["message"]
   └─ tools.byProvider[openai].deny: ["exec"]

5. Agent-specific Tool Policy（Agent级策略）
   └─ agents.list[].tools.allow: ["read"]
   └─ agents.list[].tools.deny: ["exec", "write"]

6. Agent Provider Policy（Agent Provider级策略）
   └─ agents.list[].tools.byProvider[openai].allow: []
   └─ agents.list[].tools.byProvider[openai].deny: ["message"]

7. Sandbox Tool Policy（沙箱工具策略）
   └─ tools.sandbox.tools.allow: ["read"]
   └─ tools.sandbox.tools.deny: ["exec"]
   └─ agents.list[].tools.sandbox.tools: {}

最终结果：[read]
```

**工具组（Tool Groups）速查表**：

| 组名 | 包含工具 | 用途 |
|-----|---------|------|
| `group:runtime` | exec, bash, process | 运行时操作 |
| `group:fs` | read, write, edit, apply_patch | 文件系统操作 |
| `group:sessions` | sessions_list, sessions_history, sessions_send | 会话管理 |
| `group:memory` | memory_search, memory_get | 记忆操作 |
| `group:ui` | browser, canvas | UI操作 |
| `group:automation` | cron, gateway | 自动化操作 |
| `group:messaging` | message | 消息发送 |
| `group:nodes` | nodes | 节点管理 |
| `group:HexaCore` | 所有HexaCore内置工具 | 完整工具集 |

### 7.5 Elevated Mode（权限提升模式）

**工作原理**：
```
Elevated Mode = 基于发送者的权限白名单

场景：
- 某些工具（如exec、write）非常危险
- 只有可信的发送者可以触发
- 其他发送者使用时需要审批或拒绝

配置：
{
  "tools": {
    "elevated": {
      "enabled": true,
      "allow": ["+86138xxxxxxx"],  // 可信发送者白名单
      "tools": ["exec", "write"]   // 需要提升权限的工具
    }
  }
}

效果：
- 白名单用户：直接使用exec/write
- 非白名单用户：需要审批或拒绝
```

**配置示例**：
```json

    }
  },
  "agents": {
    "list": [
      {
        "id": "restricted",
```

| **高安全要求** | 沙箱层多层隔离 | 极高 |
| **企业级部署** | 六元组全覆盖，开箱即用 | 高 |

### 10.2 不推荐使用场景

| 场景 | 原因 | 推荐方案 |
|-----|------|---------|
| **单渠道、单Agent** | 架构过度设计 | 单Agent模式 |
| **快速原型验证** | 配置复杂度高 | 简化版架构 |
| **低频使用（<100次/天）** | 记忆层价值不大 | 关闭记忆层 |
| **无安全要求** | 沙箱层不必要 | 关闭沙箱层 |

## 十二、常见问题与最佳实践

### 12.1 常见问题

**Q1: 如何调试路由规则？**
```bash
# 测试路由
HexaCore routing test \
  --channel whatsapp \
  --accountId business \
  --peer +86138xxxxxxx \
  --content "报价"

# 查看路由日志
HexaCore logs --module routing
```

**Q2: 如何迁移现有Agent？**
```bash
# 导出Agent配置
HexaCore agents export vip > vip-config.json

# 在新环境导入
HexaCore agents import vip-config.json
```

**Q3: 如何监控Agent性能？**
```bash
# 查看Agent统计
HexaCore agents stats vip

# 查看Session执行时间
HexaCore sessions stats --agent vip --sort duration
```

**Q4: 如何回滚配置？**
```bash
# 查看配置历史
HexaCore config history

# 回滚到指定版本
HexaCore config rollback 2026-02-27T20:00:00Z
```

**Q5: 如何优化成本？**
```bash
# 启用Compaction
# 在配置中设置：
{
  "compaction": {
    "mode": "auto",
    "targetTokens": 150000
  }
}

# 启用Session Pruning
# 在配置中设置：
{
  "contextPruning": {
    "mode": "cache-ttl",
    "ttl": "5m"
  }
}

# 查看成本分析
HexaCore sessions stats --sort tokens --limit 20
```

### 12.2 最佳实践

**实践1：分层配置权限**
- 全局默认：宽松配置
- Agent级：逐步收缩
- 最小权限原则

**实践2：合理设置压缩阈值**
- 避免过早压缩：过早会丢失上下文
- 避免过晚压缩：过晚会超出窗口
- 推荐值：150000 tokens（200K窗口的75%）

**实践3：利用TTL优化缓存**
- 高并发场景：短TTL（5-10分钟）
- 低并发场景：长TTL（30-60分钟）

**实践4：沙箱模式选择**
- 个人助手：mode "off"
- 家庭助手：mode "all", scope "agent"
- 公开助手：mode "all", scope "session"

**实践5：定期备份配置**
```bash
# 导出完整配置
HexaCore config export > backup-$(date +%Y%m%d).json

# 备份工作区
tar -czf workspace-$(date +%Y%m%d).tar.gz ~/.HexaCore/workspace-*
```

**实践6：监控和告警**
```bash
# 监控Gateway状态
watch -n 5 'HexaCore gateway status'

# 监控日志
tail -f ~/.HexaCore/logs/gateway.log | grep -E "error|warning"

# 监控成本
HexaCore sessions stats --sort tokens --limit 10
```

## 十三、总结与展望

### 13.1 六元组架构的核心价值

**解耦**：
- 出入口、路由、频道、技能、记忆、沙箱各司其职，互不干扰
- 每个模块都可以独立测试、独立扩展、独立部署

**扩展**：
- 新增渠道、规则、Agent、Skill、记忆规则、权限策略只需配置
- 无需修改核心代码

**隔离**：
- 不同Agent的工作区、会话、权限完全隔离
- 容器化部署，故障不扩散

**可观测**：
- 通过Session Key、审计日志、性能监控全面掌控系统状态

**优化**：
- Compaction自动压缩，成本降低96%
- Session Pruning优化缓存，响应时间降低68%

### 13.2 关键成功因素

| 🔒 严格的权限控制 | 遵循最小权限原则 |
| 📊 完善的监控体系 | 建立日志、审计、性能监控 |
| 💰 持续的成本优化 | 启用Compaction和Pruning |
