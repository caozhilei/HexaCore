# 基于六元组架构的自主进化智能体工作平台 - 测试计划 v2.0

## 1. 概述

### 1.1 项目背景
基于六元组架构（出入口、路由、频道、技能、记忆、沙箱）构建的生产级自主进化智能体工作平台，旨在支持多渠道接入、智能路由分发、上下文隔离、技能热插拔、记忆优化与安全沙箱，实现复杂任务成功率>85%和平均响应时间<5秒的核心性能指标。

### 1.2 测试目的
本测试计划旨在确保六元组架构各层功能正确性、接口兼容性、性能达标性和安全可靠性，为平台上线提供质量保障。通过系统化测试验证，确保：
- 各元组核心模块按设计要求正确实现
- 六个元组间的数据流转和接口调用无异常
- 核心性能指标（成功率、响应时间、Token成本优化）达到预期
- 沙箱安全隔离机制有效防止权限逃逸和资源滥用

### 1.3 文档结构
- 第2章：测试策略 - 整体测试方法、范围、资源需求
- 第3章：测试类型与覆盖范围 - 详细测试分类和覆盖目标
- 第4章：测试用例设计 - 各类型测试用例示例

- 第7章：验收标准 - 各测试类型通过标准
- 第8章：风险分析与应对措施

## 2. 测试策略

### 2.1 测试目标
1. **功能正确性**：验证每个元组独立功能和六个元组协同工作的业务逻辑正确
2. **接口完整性**：确保各层间接口协议兼容，数据格式转换无误
3. **性能达标性**：验证复杂任务成功率>85%、平均响应时间<5秒、Token成本优化-96%
4. **安全可靠性**：验证沙箱三层防护机制有效，无高危安全漏洞
5. **可维护性**：确保代码质量达标，单元测试覆盖率>85%

### 2.2 测试范围
| 测试范围 | 包含内容 | 排除内容 |
|----------|----------|----------|
| **出入口层** | WhatsApp/企业微信/Web适配器、协议转换、InboundMessage标准化 | 第三方渠道SDK内部实现 |
| **路由层** | 7级匹配规则引擎、智能分发、优先级调度 | 外部规则存储服务 |
| **频道层** | Session Key机制、会话隔离、工作区管理、Redis集成 | Redis集群高可用性 |
| **技能层** | Skill声明式定义、热插拔机制、权限解耦、沙箱执行 | 第三方技能市场对接 |
| **记忆层** | SQLite存储、向量检索、Compaction、Pruning、Memory Sidecar | OpenAI Embedding服务稳定性 |
| **沙箱层** | Docker容器隔离、工具权限过滤、权限提升管理、审计日志 | Docker引擎安全加固 |
| **集成工作流** | 企业级智能客服示例全链路、数据流转验证 | 实际生产环境部署 |

### 2.3 测试方法
1. **自底向上测试**：从单元测试开始，逐步进行集成测试、系统测试
2. **分层隔离测试**：每个元组独立测试，再验证层间接口
3. **黑盒与白盒结合**：功能测试使用黑盒方法，性能和安全测试使用白盒方法
4. **自动化优先**：单元测试和集成测试实现100%自动化，性能测试半自动化
5. **持续集成**：测试用例集成到CI/CD流水线，每次代码变更自动执行

### 2.4 资源需求
| 资源类型 | 规格要求 | 数量 | 用途 |
|----------|----------|------|------|
| **开发环境** | Node.js 18+、TypeScript 5+、Docker 24+ | 1套 | 开发与单元测试 |
| **测试环境** | 4核8G内存、100GB SSD、Docker Compose | 1套 | 集成测试与性能测试 |
| **数据库** | SQLite（嵌入式）、Redis 7+ | 各1 | 记忆层与频道层测试 |
| **网络环境** | 可访问OpenAI API、稳定外网连接 | - | 向量检索测试 |
| **测试工具** | Jest、Supertest、k6、OWASP ZAP | - | 各类测试执行 |
| **人力资源** | 测试工程师1人、开发工程师1人 | 2人 | 测试设计、执行与问题跟踪 |

### 2.5 风险分析
| 风险项 | 概率 | 影响 | 缓解措施 |
|--------|------|------|----------|
| **性能指标不达标** | 中 | 高 | 早期性能基准测试，优化关键路径，预留性能调优时间 |
| **接口兼容性问题** | 中 | 中 | 制定严格的接口契约测试，使用TypeScript接口类型检查 |
| **安全漏洞遗漏** | 低 | 高 | 采用多层安全测试，包括静态分析、动态扫描和手动渗透测试 |
| **测试环境不一致** | 高 | 中 | 使用Docker容器化环境，确保环境一致性 |
| **第三方服务不稳定** | 中 | 中 | 模拟外部服务，使用Mock和Stub减少依赖 |

## 3. 测试类型与覆盖范围

### 3.1 单元测试
**测试对象**：每个元组的核心模块独立功能
**覆盖目标**：
- 代码行覆盖率 >85%
- 分支覆盖率 >80%
- 函数覆盖率 >90%
**测试重点**：
- 出入口适配器的协议转换正确性
- 路由引擎的7级匹配规则逻辑
- 频道管理器的会话生命周期管理
- 技能框架的加载、注册、执行流程
- 记忆层的Compaction、Pruning算法正确性
- 沙箱层的工具权限过滤逻辑

### 3.2 集成测试
**测试对象**：六个元组间的接口和数据流转
**覆盖目标**：
- 端到端场景通过率 >95%
- 接口调用成功率 100%
- 数据格式转换正确性 100%
**测试重点**：
- 出入口→路由：消息格式正确传递
- 路由→频道：会话正确路由
- 频道→技能：上下文正确传递
- 技能→沙箱：安全执行环境
- 记忆层→各层：数据正确存储和检索
- 企业级智能客服示例全链路验证

### 3.3 性能测试
**测试对象**：核心性能指标验证
**覆盖目标**：
- 复杂任务成功率 >85%
- 平均响应时间 <5秒
- Token成本优化 -96%
- 系统吞吐量：支持100并发用户
**测试重点**：
- 单用户典型场景响应时间
- 并发用户负载下的成功率
- Token消耗监控与优化验证
- 内存和CPU资源使用率
- 数据库查询性能

### 3.4 安全测试
**测试对象**：沙箱层安全隔离机制
**覆盖目标**：
- 无高危漏洞（CVSS评分 >=7.0）
- 权限逃逸防护成功率 100%
- 审计日志完整性 100%
**测试重点**：
- Docker容器逃逸防护
- 7层工具权限过滤有效性
- 权限提升管理安全性
- 输入验证和输出编码
- 敏感信息泄露防护

### 5.5 问题跟踪流程
1. **缺陷发现**：测试执行中发现的问题记录到问题跟踪系统
2. **严重等级划分**：
   - P0：阻塞性问题，必须立即修复
   - P1：重要功能缺陷，影响核心指标
   - P2：一般功能缺陷，影响用户体验
   - P3：轻微问题，建议优化
3. **修复与验证**：开发修复后，测试重新执行相关用例
4. **回归测试**：每个版本发布前执行全量回归测试

```javascript
import http from 'k6/http';
import { check, sleep } from 'k6';
import { Rate } from 'k6/metrics';

export const errorRate = new Rate('errors');

export const options = {
  stages: [
    { duration: '2m', target: 50 },    // 逐步增加到50并发
    { duration: '5m', target: 50 },    // 保持50并发5分钟
    { duration: '2m', target: 100 },   // 增加到100并发
    { duration: '3m', target: 100 },   // 保持100并发3分钟
    { duration: '2m', target: 0 },     // 逐步降为0
  ],
  thresholds: {
    http_req_duration: ['p(95)<5000'], // 95%请求<5秒
    errors: ['rate<0.15'],             // 错误率<15%
  },
};

export default function () {
  const response = http.post('http://localhost:3000/wecom/webhook', {
    // 测试数据
  });
  
  const success = check(response, {
    'status is 200': (r) => r.status === 200,
    'response time < 5s': (r) => r.timings.duration < 5000,
  });
  
  errorRate.add(!success);
  sleep(1);
}
```
