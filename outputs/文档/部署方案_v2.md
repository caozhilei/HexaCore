# 基于六元组架构的自主进化智能体工作平台 部署方案与运维监控策略_v2

## 1. 概述

### 1.1 文档目的
本文档基于六元组架构（出入口、路由、频道、技能、记忆、沙箱）技术方案文档集、核心模块示例代码、企业级智能客服系统示例和测试计划_v2，制定完整的生产级部署方案和运维监控策略，为开发、测试、生产多环境部署提供具体可行的技术指导。

### 1.2 适用范围
- **目标架构**：基于HexaCore框架的六元组架构
- **部署环境**：开发环境（Docker Compose）、测试环境（K8s集群）、生产环境（多可用区K8s集群）
- **技术栈**：Node.js + TypeScript + SQLite + Redis + Docker + Kubernetes
- **运维体系**：监控告警、故障处理、版本发布、安全加固、性能调优全流程

### 1.3 与六元组架构的集成关系
部署方案完全继承并落地前序设计文档的技术选型和架构决策：

| 六元组元组 | 关键部署特性 | 部署实现 |
|-----------|------------|---------|
| **出入口层** | 多渠道协议适配、安全DM策略、异步消息处理 | 多Deployment部署，支持WhatsApp、企业微信、Web渠道独立扩缩容 |
| **路由层** | 7级匹配规则、无状态路由引擎 | 无状态服务，水平扩展，负载均衡 |
| **频道层** | Session Key机制、上下文隔离 | Redis集群会话存储，频道管理器高可用 |
| **技能层** | 声明式Skill定义、热插拔机制 | 技能注册中心，沙箱容器隔离 |
| **记忆层** | SQLite向量检索、Compaction算法、Memory Sidecar | 持久化存储卷，Compaction定时任务，Sidecar容器部署 |
| **沙箱层** | 容器隔离、7层工具权限过滤、权限提升管理 | 专用沙箱命名空间，安全策略配置，审计日志 |

## 2. 部署架构设计

### 2.1 环境划分策略

| 环境 | 用途 | 部署方案 | 网络隔离 | 数据策略 |
|------|------|---------|---------|---------|
| **开发环境** | 功能开发、本地调试 | Docker Compose单节点 | 开发VPN | 模拟数据，每日清理 |
| **测试环境** | 集成测试、性能压测 | 3节点K8s集群 | 测试VPC | 生产数据脱敏副本 |

| **生产环境** | 线上服务 | 多可用区K8s集群，高可用 | 生产VPC | 生产数据，严格备份 |

### 2.2 服务器配置规格

#### 2.2.1 Kubernetes集群节点规划

**控制平面节点**（3节点高可用，跨可用区）：
- 规格：4核CPU / 16GB内存 / 100GB SSD
- 角色：etcd、kube-apiserver、kube-controller-manager、kube-scheduler
- 部署方式：专用节点，不运行工作负载

**工作节点**（弹性扩展组）：
- **常规计算节点**：8核CPU / 32GB内存 / 200GB SSD
  - 运行业务微服务（出入口适配器、路由引擎、频道管理器等）
- **内存优化节点**：16核CPU / 64GB内存 / 200GB SSD
  - 运行Redis集群、会话状态服务
- **存储优化节点**：8核CPU / 32GB内存 / 1TB SSD + 2TB HDD
  - 运行SQLite数据库、记忆层持久化存储

### 2.3 负载均衡方案

#### 2.3.1 外部负载均衡
- **云服务商LB**：TCP/UDP四层负载，负责将流量分发到不同可用区的Ingress节点
- **健康检查**：HTTP GET /health，间隔10秒，失败阈值2次
- **会话保持**：基于源IP的会话保持，超时时间30分钟

#### 2.3.2 内部负载均衡
- **Kubernetes Service**：ClusterIP类型，服务发现基础
- **入口流量分发**：基于路径的路由规则，将不同渠道流量分发到对应的出入口适配器
- **负载均衡算法**：轮询（Round Robin）为主，支持最少连接（Least Connection）

## 3. 容器化部署方案

### 3.1 Docker镜像构建规范

#### 3.1.1 基础镜像选择
```dockerfile
# 基础镜像：官方Node.js LTS镜像，指定版本
FROM node:20-slim

# 设置时区和语言
ENV TZ=Asia/Shanghai
RUN ln -snf /usr/share/zoneinfo/$TZ /etc/localtime && echo $TZ > /etc/timezone
ENV LANG C.UTF-8

# 安装系统依赖（包括SQLite开发包）
RUN apt-get update && apt-get install -y \
    gcc \
    g++ \
    python3 \
    make \
    sqlite3 \
    libsqlite3-dev \
    curl \
    && rm -rf /var/lib/apt/lists/*

# 创建工作目录
WORKDIR /app

# 复制package文件
COPY package*.json ./

# 安装依赖（生产环境）
RUN npm ci --only=production

# 复制源代码
COPY . .

# 构建TypeScript代码
RUN npm run build

# 非root用户运行
RUN useradd -m -u 1000 appuser && chown -R appuser:appuser /app
USER appuser

# 健康检查
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD curl -f http://localhost:3000/health || exit 1

# 启动命令
CMD ["node", "dist/index.js"]
```

  name: memory-compaction
  namespace: agent-platform

  schedule: "0 */6 * * *"  # 每6小时执行一次
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: compaction-worker
            image: registry.example.com/agent-platform/memory-compactor:v1.0.0
            env:
            - name: COMPACTION_THRESHOLD
              value: "0.85"  # 压缩阈值85%

              requests:

                cpu: "500m"
              limits:
                memory: "2Gi"
                cpu: "1000m"

### 8.2 故障处理预案

#### 8.2.1 故障分级与响应时间

| 故障级别 | 定义 | 响应时间 | 处理时限 | 升级路径 |
|---------|------|---------|---------|---------|

### 8.3 容量规划与扩缩容策略

#### 8.3.1 六元组容量评估模型

**基础容量公式**：
```
总业务负载 = 峰值QPS × 平均处理时间 × 安全系数(1.5)

出入口层容量 = 总业务负载 / (单适配器处理能力 × 渠道数)
路由层容量 = 总业务负载 / (单路由引擎处理能力)
频道层容量 = 活跃会话数 × 会话内存占用 / Redis节点内存
记忆层容量 = 日增记忆数据量 × 保留天数 / 单节点存储
```

**当前平台容量基准**：
- 峰值QPS：500请求/秒
- 平均处理时间：3秒
- 活跃会话数：10,000
- 日增记忆数据：10GB
- 记忆保留天数：30天

### 12.3 常见问题排查指南

**问题1：出入口层消息丢失**
- 检查适配器连接状态
- 验证消息队列配置
- 检查网络防火墙规则

**问题2：路由匹配结果异常**
- 检查路由规则配置文件
- 验证7级匹配逻辑
- 查看路由决策日志

**问题3：记忆压缩率不达标**
- 检查Compaction服务运行状态
- 分析记忆数据分布特征
- 调整压缩算法参数
- 验证Embedding模型准确性

**问题4：沙箱工具执行失败**
- 检查工具权限策略
- 验证沙箱容器资源限制
- 查看安全审计日志
- 排查网络代理配置

**问题5：性能指标不达标**
- 分析各层监控数据
- 定位性能瓶颈层
- 调整对应调优参数
- 考虑硬件资源升级
