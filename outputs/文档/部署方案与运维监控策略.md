## 2. 部署架构设计

### 2.1 环境划分策略

| 环境 | 用途 | 部署规模 | 网络隔离 | 数据策略 |
|------|------|---------|---------|---------|
| **开发环境** | 功能开发、本地调试 | 单节点Minikube | 开发VPN | 模拟数据，每日清理 |
| **测试环境** | 集成测试、性能压测 | 3节点K8s集群 | 测试VPC | 生产数据脱敏副本 |
| **预发环境** | 功能验证、灰度测试 | 与生产相同架构 | 预发VPC | 生产数据只读副本 |
| **生产环境** | 线上服务 | 多可用区K8s集群 | 生产VPC | 生产数据，严格备份 |

### 2.2 服务器配置规格

#### 2.2.1 Kubernetes集群节点规划

**控制平面节点**（3节点高可用）：
- 规格：4核CPU / 16GB内存 / 100GB SSD
- 角色：etcd、kube-apiserver、kube-controller-manager、kube-scheduler
- 部署方式：专用节点，不运行工作负载

**工作节点**（弹性扩展组）：
- **常规计算节点**：8核CPU / 32GB内存 / 200GB SSD
  - 运行业务微服务（API网关、决策引擎、执行层等）
- **大数据节点**：16核CPU / 64GB内存 / 1TB SSD + 2TB HDD
  - 运行数据密集型服务（InfluxDB、PostgreSQL、Spark）
- **GPU节点**：8核CPU / 32GB内存 / 1x NVIDIA A10G / 200GB SSD
  - 运行机器学习服务（Ray RLlib、TorchServe）

#### 2.2.2 网络拓扑设计

```
互联网用户 → 云服务商LB（TCP/UDP负载均衡）
               ↓
        Kubernetes Ingress Controller（Nginx）
               ↓
    ┌─────────────┬─────────────┐
    │             │             │
 API网关服务   认证服务    监控服务
    │             │             │
    └─────────────┴─────────────┘
               ↓
    ┌───────────────────────────┐
    │     服务网格（Istio）      │
    │ 流量管理/安全/可观测性     │
    └───────────────────────────┘
               ↓
    ┌─────────────┬─────────────┐
    │             │             │
 决策层服务    执行层服务   进化引擎服务
    │             │             │
    └─────────────┴─────────────┘
               ↓
    ┌───────────────────────────┐
    │     数据存储层             │
    │ PostgreSQL/Redis/InfluxDB │
    │   RabbitMQ/Elasticsearch  │
    └───────────────────────────┘
```

### 2.3 负载均衡方案

#### 2.3.1 外部负载均衡
- **云服务商LB**：TCP/UDP四层负载，负责将流量分发到不同可用区的Ingress节点
- **健康检查**：HTTP GET /health，间隔10秒，失败阈值2次
- **会话保持**：基于源IP的会话保持，超时时间30分钟

#### 2.3.2 内部负载均衡
- **Kubernetes Service**：ClusterIP类型，服务发现基础
- **Istio VirtualService**：七层流量管理，支持金丝雀发布、故障注入
- **负载均衡算法**：轮询（Round Robin）为主，支持最少连接（Least Connection）

## 3. 容器化部署方案

### 3.1 Docker镜像构建规范

#### 3.1.1 基础镜像选择
```dockerfile
# 基础镜像：官方Python镜像，指定版本
FROM python:3.9-slim

# 设置时区和语言
ENV TZ=Asia/Shanghai
RUN ln -snf /usr/share/zoneinfo/$TZ /etc/localtime && echo $TZ > /etc/timezone
ENV LANG C.UTF-8

# 安装系统依赖
RUN apt-get update && apt-get install -y \
    gcc \
    g++ \
    curl \
    && rm -rf /var/lib/apt/lists/*
```

#### 3.1.2 多阶段构建示例（以API网关为例）
```dockerfile
# 构建阶段
FROM python:3.9-slim as builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --user --no-cache-dir -r requirements.txt

# 运行阶段
FROM python:3.9-slim
WORKDIR /app
COPY --from=builder /root/.local /root/.local
COPY . .

# 非root用户运行
RUN useradd -m -u 1000 appuser && chown -R appuser:appuser /app
USER appuser

# 健康检查
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD curl -f http://localhost:8000/health || exit 1

# 启动命令
CMD ["uvicorn", "api_gateway.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

## 4. 数据库与中间件部署

### 4.1 PostgreSQL集群配置

#### 4.1.1 高可用架构
采用Patroni + etcd + PgBouncer架构：
- **Primary + 2 Standby**：跨可用区部署，自动故障切换
- **同步复制**：至少一个同步备节点，确保数据零丢失
- **连接池**：PgBouncer连接池，最大连接数1000

#### 4.1.2 关键配置参数
```yaml
# postgresql.conf关键配置
max_connections = 500
shared_buffers = 8GB
work_mem = 32MB
maintenance_work_mem = 1GB
wal_level = logical
max_wal_senders = 10
synchronous_commit = remote_apply
synchronous_standby_names = 'ANY 1 (standby1, standby2)'

# pg_hba.conf配置
host    all             all             10.0.0.0/8            md5
host    replication     all             10.0.0.0/8            md5
```

#### 4.1.3 备份恢复策略
- **每日全量备份**：凌晨2点，保留30天
- **连续WAL归档**：归档到S3对象存储
- **恢复点目标（RPO）**：< 5分钟
- **恢复时间目标（RTO）**：< 30分钟
- **备份验证**：每周自动恢复验证

### 4.2 Redis集群部署

#### 4.2.1 Redis Sentinel高可用
- **3主3从**：每个主节点配一个从节点
- **Sentinel集群**：3节点Sentinel，负责监控和自动故障转移
- **内存配置**：每个实例最大内存16GB，启用LRU淘汰策略

#### 4.2.2 持久化配置
```conf
# redis.conf关键配置
appendonly yes
appendfsync everysec
save 900 1
save 300 10
save 60 10000
maxmemory 16gb
maxmemory-policy volatile-lru
```

#### 4.2.3 缓存预热策略
- 服务启动时加载热点数据
- 定时任务更新缓存
- 缓存穿透防护：布隆过滤器

### 4.3 RabbitMQ集群配置

#### 4.3.1 镜像队列高可用
- **3节点集群**：跨可用区部署
- **镜像队列**：所有队列镜像到至少2个节点
- **负载均衡**：客户端连接轮询所有节点

#### 4.3.2 关键配置
```yaml
# rabbitmq.conf
loopback_users.guest = false
default_user = platform_user
default_pass = ${RABBITMQ_PASSWORD}
vm_memory_high_watermark.relative = 0.7
disk_free_limit.relative = 2.0
cluster_formation.peer_discovery_backend = rabbit_peer_discovery_k8s
cluster_formation.k8s.host = kubernetes.default.svc.cluster.local
```

#### 4.3.3 消息持久化
- 持久化消息：delivery_mode=2
- 队列持久化：durable=true
- 发布确认：publisher confirms
- 消费确认：手动ack

### 5.4 告警规则配置

#### 5.4.1 关键告警规则（继承自性能监控模块设计）
```yaml
# rules/critical.yml
groups:
- name: critical_alerts
  rules:
  - alert: ComplexTaskSuccessRateBelow85
    expr: avg_over_time(agent_task_success_rate{task_type="complex"}[10m]) < 0.85
    for: 5m
    labels:
      severity: critical
      component: performance
    annotations:
      summary: "复杂任务成功率低于85%"
      description: "复杂任务成功率当前为{{ $value }}，低于目标值85%，可能影响平台核心指标"
  
  - alert: AvgResponseTimeAbove5s
    expr: avg_over_time(agent_task_response_time_avg[5m]) > 5000
    for: 3m
    labels:
      severity: warning
      component: performance
    annotations:
      summary: "平均响应时间超过5秒"
      description: "平台平均响应时间当前为{{ $value }}ms，超过目标值5秒"
  
  - alert: DatabaseConnectionPoolExhausted
    expr: postgresql_connection_pool_usage > 0.9
    for: 2m
    labels:
      severity: critical
      component: database
    annotations:
      summary: "数据库连接池使用率超过90%"
      description: "PostgreSQL连接池使用率{{ $value }}，可能影响业务"
```

#### 6.1.3 回滚机制
- **自动回滚条件**：
  - 错误率>5%持续5分钟
  - 平均响应时间>8秒持续5分钟
  - 服务健康检查失败持续3次
- **手动回滚命令**：
  ```bash
  kubectl rollout undo deployment/api-gateway -n agent-platform
  ```

### 6.2 故障处理预案

#### 6.2.1 故障分级与响应时间

| 故障级别 | 定义 | 响应时间 | 处理时限 | 升级路径 |
|---------|------|---------|---------|---------|
| **P0紧急** | 核心功能完全不可用，影响所有用户 | <5分钟 | <30分钟 | 技术总监→CTO |
| **P1严重** | 核心功能部分不可用，影响>30%用户 | <15分钟 | <2小时 | 运维经理→技术总监 |
| **P2一般** | 非核心功能异常，影响<30%用户 | <1小时 | <8小时 | 值班工程师→运维经理 |
| **P3轻微** | 性能下降，不影响功能使用 | <4小时 | <24小时 | 值班工程师处理 |

#### 6.2.2 常见故障处理流程

**场景1：数据库连接池耗尽**
1. 立即扩容数据库连接池参数
2. 分析慢查询，优化SQL
3. 考虑读写分离、分库分表
4. 长期：数据库垂直/水平拆分

**场景2：内存泄漏导致Pod频繁重启**
1. 查看内存使用趋势，定位泄漏服务
2. 临时增加Pod内存限制
3. 分析堆内存dump，修复代码
4. 重启有问题的服务实例

**场景3：网络分区导致服务不可用**
1. 检查Kubernetes节点状态
2. 重启故障节点的kubelet
3. 临时将流量切换到其他可用区
4. 网络设备排查修复

### 6.3 容量规划与扩缩容策略

#### 6.3.1 容量评估模型

**基础容量公式**：
```
所需节点数 = ceil(总业务负载 / 单节点容量)

总业务负载 = 峰值QPS × 平均处理时间 × 安全系数(1.5)
单节点容量 = (节点CPU核数 × CPU利用率目标) / (单请求CPU消耗)
```

**当前平台容量基准**：
- 峰值QPS：1000请求/秒
- 平均处理时间：3秒
- 单请求CPU消耗：0.1核
- 节点规格：8核32GB
- CPU利用率目标：70%

计算结果：所需节点数 = ceil((1000 × 3 × 1.5) / (8 × 0.7 / 0.1)) = ceil(4500 / 56) ≈ 81节点

#### 6.3.2 自动扩缩容配置

**水平Pod自动扩缩容（HPA）**：
```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-gateway-hpa
  namespace: agent-platform
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-gateway
  minReplicas: 3
  maxReplicas: 50
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80
  - type: Pods
    pods:
      metric:
        name: http_requests_per_second
        target:
          type: AverageValue
          averageValue: 100
```

**集群节点自动扩缩容**：
- 触发条件：集群资源请求>80%持续10分钟
- 扩容策略：每次增加2个工作节点
- 缩容策略：资源使用率<30%持续30分钟，减少1个节点

#### 6.3.3 容量监控预警
- **黄色预警**：资源使用率>70%持续1小时
- **橙色预警**：资源使用率>80%持续30分钟
- **红色预警**：资源使用率>90%持续15分钟
